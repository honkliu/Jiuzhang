# 管理员帮助

## 聊天

管理员主要负责用户和域群的基础秩序。缺省域群来自邮箱域名，所以用户邮箱域要保持一致。

管理检查:

- 先确认用户邮箱域，再检查缺省域群。
- 洛是系统用户，应由服务器启动流程初始化。
- 如果清空过数据库，重启服务器让初始化流程重新执行。

聊天命令:

- `/h` 显示聊天命令帮助。
- `/w` 查看聊天成员。
- `/wa` 查看当前活跃成员。
- `/b <文字>`、`/i <文字>`、`/r <文字>` 分别发送粗体、斜体、红色文字。
- `/p [提示词]` 生成图片；群聊里可用 `@某人` 指定人物。
- `/a @某人` 把某人加入当前聊天。
- `@名字` 提及某个人；`@@` 呼叫洛。

2D 沉浸式聊天管理要点:

- 入口在一对一聊天标题栏右侧的 2D 按钮。
- 用户进入后仍然发送普通一对一消息，只是左右分屏展示最近内容。
- 如果用户看不到 2D 按钮，先确认当前是一对一聊天。
- 如果显示不对，先检查该一对一聊天的消息数据和前端控制台错误。

示例:

- `amy@shaol.com` 属于 `shaol.com` 缺省域群。

## bbtalk

bbtalk 给需要终端聊天的人使用，也适合检查 Wa 的长回答、Markdown、表格和公式显示。

bbtalk 管理要点:

- 安装目录是 `KanKan/bbtalk`，首次使用运行 `npm install`。
- 登录命令是 `node index.js login <email> <password> --base-url <api-base-url>`。
- 启动命令是 `npm start` 或 `node index.js`。
- 如果无法连接，先检查 API base URL、登录 token、KanKan server 和 SignalR 连接。

bbtalk 命令:

- 聊天: `/cl` 列聊天，`/cj <编号>` 进聊天，`/cq` 离开当前聊天，`/cd` 清空/删除当前聊天记录，`/cn` 显示当前聊天名称。
- 联系人: `/ul` 列用户，`/ua <编号>` 发好友请求，`/c <编号>` 创建一对一聊天，`/ur` 看请求，`/urc [编号]` 接受，`/urd [编号]` 拒绝。
- 管理员: `/du <编号>` 删除 `/ul` 列表中的用户。
- 其他: `/help` 显示帮助，`/quit` 退出程序。

## 图片美化

图片美化是一个主要功能，依赖图片上传、提示词组合、生成任务以及生成结果保存。

管理员要点:

- 用户可从聊天、啪、家谱、票夹和图览图片发起生成。
- 提示词库内容应短、可复用。
- 生成结果会挂在原图下，便于比较原图和美化图。
- 如果生成慢，先查图片生成服务，不要急着改 UI。

示例:

- 好提示词如 `formal portrait, clean background`，弱提示词如 `make it better`。

## 札记

需要长期保存的共享知识，放在 **札记** 比聊天更合适。页面标题尽量短，大主题拆成分区。

管理:

- 用设置管理谁能查看或编辑札记。
- 大清理前先导出重要札记。
- 导入可用于恢复或迁移札记归档。
- 建议一个主题一个札记本，避免所有内容塞进一本。

示例:

- 可创建 `项目`、`家族资料`、`决策记录` 等分区。

## 家谱

管理员为已开通的域创建、导入和管理家谱。建议先维护一棵干净的主树，再补充人物、关系和谱志页面。

设置:

- 先开通域名，用户才会看到家谱入口。
- 创建或导入后，检查根人物、姓氏和辈分。
- 跨支系共享时，按用户或域设置可见权限。

关系:

- 父母、配偶、子女决定树的主体结构。
- `出继` 只在谱系含义明确时使用。
- 关联人物连接不同家谱中的同一人，但不会合并两棵树。

示例:

- 为 `shaol.com` 创建家谱，导入根人物，再添加父母、配偶和子女。
- 把支谱祖先关联回主谱中的同一祖先。

## 票夹

管理员通常协助整理：重复照片、错误日期、识别错误。目标是可检索记录，不是追求完美 OCR。

检查清单:

- 检查日期、总金额、商家或医院名。
- 尽量保留原图，它们是核对依据。
- 用图片集合做视觉整理。

示例:

- 批量识别后，先抽查金额和日期，再使用统计结果。

## 啪

**啪** 基本不需要复杂管理。保持氛围友好，只处理明显不合适的内容即可。

管理提示:

- 如果某条内容值得长期保存，把它整理到札记里，不要只留在啪里。
